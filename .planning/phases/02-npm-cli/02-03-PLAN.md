Create packages/npm/src/utils/detect-python.ts:

Export detectPython(): Promise<string | null>

Implementation: probe ['python3', 'python', 'py'] in order using execa. For each:
  - Try execa(cmd, ['--version'])
  - On ENOENT (ExecaError code === 'ENOENT'): continue to next
  - On success: parse stdout (and stderr — python < 3.4 prints version to stderr) with regex /Python (\d+)\.(\d+)/
  - If no match: continue (Windows Store guard — Store stub may return empty or non-version output)
  - If major > 3: return cmd (future Python 4 compatibility)
  - If major === 3 and minor >= 10: return cmd
  - If major === 3 and minor < 10: continue (version too old)
  - If major < 3: continue
  After loop: return null

Note: execa v9 returns stderr in { stderr } — check both stdout and stderr for the version string since Python 2 printed to stderr.

Export installHeadroom(log: (msg: string) => void): Promise<void>

The log parameter accepts a function for printing messages — this enables testing without console.log side effects. In production use, the caller passes a function that calls the @clack/prompts output or console.error.

Implementation:
  1. Call detectPython()
  2. If null: log("Python 3.10+ not found — skipping headroom install. Install Python 3.10+ and run `goodvibes init` again."); return — DO NOT throw, DO NOT process.exit(1)

  6. If all three fail (all ENOENT): log a warning that no installer was found and return — do NOT exit 1 (graceful degrade per HDR-02 spirit)

IMPORTANT: No subprocess call uses shell: true. Args are always arrays. No string interpolation in command args.

Export configureMcp(log: (msg: string) => void): Promise<void>

Implementation:
  1. First check idempotency: try execa('headroom', ['mcp', 'status'])
     - If exit code 0: log("headroom MCP already configured — skipping"); return
     - If ENOENT or non-zero: continue to registration
  2. Try primary strategy (Strategy B — handles CLAUDE_CONFIG_DIR correctly):
     a. try execa('claude', ['mcp', 'list'])
     b. If stdout includes 'headroom': log("headroom already registered in claude MCP — skipping"); return
     c. Resolve absolute headroom path: result of execa('which', ['headroom']) on Linux/macOS, execa('where', ['headroom']) on Windows (process.platform === 'win32')
     d. execa('claude', ['mcp', 'add', '-s', 'user', 'headroom', absolutePath.trim()])
     e. log("headroom registered as global MCP server")
  3. Catch ExecaError with code ENOENT from step 2 (claude CLI not on PATH):
     - log("claude CLI not found — falling back to headroom mcp install")
     - log("Warning: if you use CLAUDE_CONFIG_DIR, you may need to run `headroom mcp install` manually")
     - execa('headroom', ['mcp', 'install'])
  4. All other unexpected errors: re-throw

IMPORTANT:

- The absolute path from `which headroom` is passed as a separate array element to execa — it is NEVER interpolated into a shell string.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI process → OS Python toolchain (uv/pipx/pip) | Subprocess execution of package manager with fixed args |
| CLI process → headroom binary | Subprocess execution of installed headroom binary |
| CLI process → claude CLI | Subprocess execution of claude CLI for MCP registration |
| headroom binary path → MCP registration | Absolute path resolved via which/where and passed to claude mcp add |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-03-A | Tampering | execa subprocess calls — command injection | mitigate | All execa calls pass args as arrays, never as shell strings. shell option is never set to true. Verified by acceptance criteria grep check. |
| T-02-03-B | Tampering | headroom absolute path from which/where — path injection | mitigate | The path returned by which headroom is from the OS PATH resolution of a just-installed binary. It is passed as a discrete execa arg element, not interpolated into a shell command. Worst case: the user's PATH has a malicious 'headroom' binary — this is the user's own machine at user privilege level, no escalation possible. |
| T-02-03-C | Tampering | install args containing "headroom-ai[all]" — bracket characters | mitigate | execa passes args as array to the process directly (no shell expansion). The '[all]' bracket is passed verbatim to uv/pipx/pip as a Python package extra, which is their standard syntax. No shell injection possible without shell:true. |
| T-02-03-D | Elevation of Privilege | pip install --user writes to user home | accept | --user flag intentionally installs to user's local site-packages (~/.local/lib), not system site-packages. No sudo or elevation required. This is the expected safe pip install pattern. |
| T-02-03-E | Information Disclosure | log() messages written to terminal | accept | Log messages contain only status text and the headroom binary path (a non-secret filesystem path). No credentials, tokens, or PII in any message. |
  T-02-03-SC | Tampering | npm/pip/cargo installs (headroom-ai[all]) | mitigate | headroom-ai is Apache 2.0 per RESEARCH.md and REQUIREMENTS.md; package verified on PyPI (headroom-ai); install is runtime, not build-time; user sees install output via execa stdio inheritance |
</threat_model>
